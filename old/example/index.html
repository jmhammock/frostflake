Sprites

Sprites are the basic object type that all game objects derive from. Sprites are not required to have a texture. Sprites can have child sprites that render in relation to the parent, taking rotation, velocity, acceleration and position from their parent sprite. This allows developers to create complex relationships from modular objects.

The example above demonstrates a parent object that subclasses the frostFlake Sprite. The parent object has no texture but contains two child objects that extend Sprite and are given positions relative to the parent. As the parent rotates, the children are rendered relative to the parent rotation with their own additional rotation applied.

Animations

Animations exist as a property on a sprite object. An animation keeps track of sequences, frames, frame durations and the current frame that should render. Animations can be defined in code but it is recommended that developers define and load animations from JSON.

This example demonstrates a spritesheet and a JSON-defined animation. Animating sprites behave just like non-animated sprites and can be attached to a parent sprite or have child sprites of their own.

Note that animations set specific sprite texture coordinates. Setting texture coordinates on a sprite also works for using sprites out of a single texture atlas!

Input

Use "R" to rotate sprite. Move mouse over canvas to move sprite. Primary mouse click to fade sprite.

frostFlake uses jQuery to track keyboard and mouse state. It also provides mappings of key and mouse button codes to human-readable names. Touchscreen input is supported but may require extra effort and testing to implement as devices are changing rapidly!

Cameras

Use WASD to move the camera.

frostFlake creates a default camera object on the game instance. The camera can have a position set or position velocity like a Sprite. You can create your own custom camera that derives from the frostFlake camera and set that as the primary camera. It's also possible to switch between different cameras simply by changing the camera used by the renderer!

In the demo above, Sprites have a "parallax" percentage property that simulates depth. The demo sprites have a random parallax percentage applied. As the camera moves, sprites appear to exist at different depths due to the applied parallax. Sprites at the edge of the screen wrap, giving the appearance of an infinite field of objects.

Note: frostFlake does not support camera rotation.

Renderers

Use WASD to move the camera.

The frostFlake renderer is flexible and extensible. In many games, you won't need to touch the renderer at all! However, it can be easily extended and customized to do some powerful stuff!

The renderer takes a camera, a target canvas and a collection of sprites for each draw call. When you add sprites to a view with the addSprite method, they will automatically be rendered to the default canvas with the default camera. In most cases, this is all a game needs. However, some games will have a large number of sprites with unchanging position and rotation relative to each other. A good example of this is the background sprites or level art in a game level. Rendering large numbers of sprites with custom translations, 60 times per second is expensive and may make your game slow and buggy. This is where the renderer really shines! It's possible to create a unqiue canvas, render a large number of sprites a single time, and then use the canvas' data URL as the texture source for a single sprite!

The above example renders 100 frost flake sprites with random rotation to a dynamic canvas (often called a render target) and then uses the dynamic canvas' dataUrl as the source for a single, large sprite. In this example, the background color of the canvas is purple but it could be any color or transparent.
